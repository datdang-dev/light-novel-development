# LLM Developer Guide: LND Studio V7.0 Architecture

This document is the **Canonical Technical Reference** for any LLM Agent (like `bmad-architect`, `bmad-dev`, `gooner-editor`, etc.) or human developer working within the LND Studio repository.

If you are asked to "Read the docs" or "Understand the architecture" before making a change, **this is what you must read.**

---

## 1. Global Concept: The "Agent as a Function" Paradigm

LND Studio V7.0 explicitly moves away from "one massive prompt" into a **microservices architecture** using specialized AI Agents. 
Agents are defined in `.agent.yaml` files under `agents/` and execute specific steps inside `core/` or `services/` via their `SKILL.md` boundaries.

### Core Tenets:
- **No Hallucination Across Boundaries**: The output of one agent is strictly checked against a JSON schema (stored in `schemas/`) before the next agent can process it.
- **Just-In-Time (JIT) Context**: Agents are intentionally "blind" until invoked. They do not know the whole novel. They only receive the exact `forensic-state.json`, `context_horizon.md`, and specific `fetish-db` rules required for the *current page*.
- **State File (`state.yaml`)**: The orchestrator maintains the progress state of every batch. **Never skip a step.** If step 4 fails out of 7, the state pauses. Rerunning resumes at step 4.

## 2. Directory Structure Deep Dive

When creating new features or fixing bugs, you must place your files in the correct directories according to the `BMAD Skill Convention`:

### `/agents` (The Personas)
Contains `.agent.yaml` definitions.
- **Rules**: Must contain `id, name, title, persona, critical_actions, menu`.
- **Key Agents**: `lnd-orchestrator` (Director K), `manga-adapter` (Kana), `lewd-writer` (Suki), `panel-forensic` (Dr. Atomic).
- **Execution**: The `menu` triggers point to `SKILL.md` paths.

### `/core` (Foundation Engines)
Heavy computational or highly complex steps.
- Examples: `/panel-forensic` (Visual OCR/Scanning), `/lewd-writer` (Prose drafting).
- Usually composed of a `SKILL.md` and a `steps/` folder for procedural execution.

### `/services` (Business Logic Pipelines)
These are orchestrator-level pipelines that combine multiple Core engines.
- Example: `/gooner-alchemist` (The master 8-step image-to-prose pipeline).
- Example: `/quality-audit` (Scores text out of 100 points based on schemas).
- Example: `/renpy-adaptation` (Script to AST to Prose).

### `/knowledge` (The "Bible" & Central Truths)
Static references used for JIT context injection.
- `/fetish-db`: Extreme adult writing tropes and terminology guidelines.
- `/sfx`: SFX translation tables (Romanized JP to VN descriptive).
- `/glossaries`: Specific lexicon data (`hentai_lexicon.md`).
- **Rule**: Agents query this folder when the Vector Delta (Physical Action + Intent) matches a specific topic.

### `/schemas` (The Enforcers)
JSON Schemas (`.schema.json`) with `additionalProperties: false`.
- Any output generated by an Agent (e.g., forensic reports, debug audits) must be structured here.

### `/docs/architecture/dynamic_design` (Sequence Diagrams)
PlantUML (`.puml`) files tracking the exact step-by-step logic of pipelines.
- Read these files to understand what agent talks to whom.

## 3. The Flagship Pipeline: `gooner-alchemist`

This is the primary automated pipeline. It is orchestrated by **Director K** (`lnd-orchestrator.agent.yaml`).

1. **`step-01-initialize`**: Reads `state.yaml`, checks missing schemas.
2. **`step-01b-context-horizon`**: Looks ahead N frames using `generate_horizon.py` to prevent timeline hallucination.
3. **`step-02-forensic-analysis`**: Delegates to Dr. Atomic. Returns a 10-section visual report (OCR, Bodily fluids, Flow).
4. **`step-03-context-loading`**: Collects the forensic report + global canon rules into a JIT compilation.
5. **`step-04-prose-generation`**: Delegates to Suki. Writes the R18 explicit prose using the requested sensory density.
6. **`step-05-quality-audit`**: Delegates to Riko. Scores the prose. Auto-rewrites if failed.
7. **`step-06-state-persistence`**: Saves to ledger.

### Discrepancy Note (V6 vs V7)
Earlier V6 architecture had ` Kana ` orchestrating the Gooner Alchemist. In V7.0, **Director K** directly manages the Gooner Alchemist and delegates directly to Atomic, Suki, and Riko. `Kana` (`manga-adapter`) is reserved for manual context generation (POC) or interactive verification loops.

## 4. How to Extend the Framework

If the User asks you to add a new Feature (e.g., an Audio-to-Prose adapter):

1. **Create the Agent**: Write a new `audio-adapter.agent.yaml` in `/agents`. Give it a persona, icon, and trigger.
2. **Create the Service**: Create `/services/audio-adaptation/`. Add a `SKILL.md` and `steps/` directory.
3. **Define Schemas**: If the output needs to be audited, create an `audio-state.schema.json` in `/schemas/`.
4. **Update Diagram**: Edit or create a `.puml` file in `/docs/architecture/dynamic_design/`.
5. **Add to Orchestrator**: Update `/agents/lnd-orchestrator.agent.yaml`'s menu to include a trigger for the new service.

**End of Guide.** Keep your tool executions strictly matching these boundaries. Use `replace_file_content` targeting specific `steps/` files for logic tweaks, and modify `knowledge/` for style adjustments.
